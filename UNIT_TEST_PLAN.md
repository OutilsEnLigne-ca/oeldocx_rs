# Unit Test Plan: `oeldocx_rs`

This document outlines the strategy for thoroughly unit-testing the WebAssembly DOCX engine. The goal is to ensure Word-level precision in parsing, state management, and document manipulation.

## 1. Required DOCX Test Files

To properly test the parser (`convert/from_docx.rs`), we need a suite of `.docx` files generated by Microsoft Word (or an equivalent standard processor). 
Please create a `tests/fixtures/` directory in this crate and populate it with the following files. 

**Ensure these files contain exactly what is described so the assertions can be deterministic.**

| File Name | Content to Include | Attributes Tested |
| :--- | :--- | :--- |
| `basic_text.docx` | 3 paragraphs. First is plain. Second has one **bold** word. Third has one *italic* and one _underlined_ word. | Basic paragraph splitting, `Run` splitting, basic booleans. |
| `formatting_heavy.docx` | Paragraphs with: Strikethrough, different Font Sizes (e.g., 10pt, 24pt), different Font Families (Arial, Times New Roman), Text Colors (Red, Blue), and Highlight Colors (Yellow). | Run properties, half-point math, hex color extraction. |
| `styles_and_headings.docx` | A Document Title, a Subtitle, Heading 1, Heading 2, and Heading 3, followed by Normal text. | Style ID extraction, `docx.styles.styles` parsing, style-to-run property cascading. |
| `lists.docx` | A 3-item bulleted list, a normal paragraph, and a 3-item numbered list. | `NumberingProperty`, `ListType` resolution, indent levels. |
| `alignment.docx` | 4 paragraphs: Left aligned, Centered, Right aligned, and Justified. | Paragraph properties, Justification enum mapping. |
| `tables.docx` | A 2x2 table with text in each cell. | Table child parsing, nested paragraph extraction, Table Row/Cell indexing. |
| `page_layout.docx` | A document with custom margins (e.g., Narrow), landscape orientation (custom width/height), and a manual Page Break. | Section properties, twips parsing, `PageBreak` block types. |

> **Note on Embedded Fonts:** The underlying `docx-rs` library currently does **not** parse `fontTable.xml` or extract embedded font files (e.g., `.odttf`) from the DOCX ZIP archive. Because of this parser limitation, the engine relies solely on extracting the `font_family` string name. The React frontend is then responsible for dynamically loading matching external web fonts (like Google Fonts) or falling back to system fonts.

## 2. Unit Test Categories

The test suite will be divided into the following modules, located alongside the code they test (e.g., inline `#[cfg(test)]` modules or inside a `tests/` integration folder).

### A. Parsing & Extraction (`src/convert/from_docx.rs`)
*   **Test:** Load `basic_text.docx`. Assert `doc.blocks.len() == 3`. Assert specific runs possess `bold == true`.
*   **Test:** Load `formatting_heavy.docx`. Assert font size conversions (e.g., `sz` of 48 in DOCX equals `font_size == 24` in our `OelRunProps`).
*   **Test:** Load `styles_and_headings.docx`. Assert `doc.styles` dictionary is populated. Assert the first paragraph has `style_id == Some("Title")`.

### B. Serialization & Round-Tripping (`src/convert/to_docx.rs`)
*   **Test:** Generate a programmatic `OelDocument` in memory, serialize it to bytes, parse it back into a new `OelDocument`, and assert `doc1 == doc2`.
*   **Test:** Load `lists.docx`, serialize it, re-parse it, and ensure bullet/numbering definitions remain intact.

### C. Command Mutations (`src/commands/*.rs`)
*   **Text Commands:** Mock a document. Call `insert_text()`. Assert the target `OelRun`'s string increases in length and cursor position shifts.
*   **Format Commands:** Mock a selection spanning multiple paragraphs. Call `apply_run_format()`. Assert the text runs are split correctly and the selected portion receives `bold = true`.
*   **Table Commands:** Mock a 2x2 table. Call `insert_row_below()`. Assert table rows length becomes 3.

### D. Controller & History (`src/controller/docx_controller.rs`)
*   **State Generation:** Test `format_at_cursor()` accurately cascades properties (Run -> Paragraph Style -> Default).
*   **Undo/Redo Stack:** Insert text, assert state. Undo, assert state reverts. Redo, assert text returns.

## 3. Adding the Tasks to VSCode

To easily run these Rust unit tests alongside your frontend tasks, add the following entry to your `vite-outilsenligne-ca-frontend/.vscode/tasks.json` array:

```json
{
  "label": "Test Rust Engine (oeldocx_rs)",
  "type": "shell",
  "command": "cargo test",
  "options": {
    "cwd": "${workspaceFolder}/../oeldocx_rs"
  },
  "group": {
    "kind": "test",
    "isDefault": true
  },
  "problemMatcher": [
    "$rustc"
  ],
  "detail": "Runs all unit and integration tests for the DOCX engine"
}
```

## 4. Execution

Once the `.docx` files are created in `oeldocx_rs/tests/fixtures/`, the tests can be executed in the terminal using:

```bash
cd ../oeldocx_rs
cargo test
```
